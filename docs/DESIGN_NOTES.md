# 設計ノート: 異種容量ディスクの冗長性と容量効率

## 背景

puddle はゾーン分割方式を採用している。異種容量ディスクを容量境界でゾーンに分割し、各ゾーンで最適な RAID を構成する。この方式には「最大容量ディスクの末尾ゾーンが孤立する」という本質的な課題がある。

## 問題

例: 2TB, 4TB, 4TB のプールに 6TB ディスクを追加した場合:

```
Zone A (0〜2TB):  4台参加 → RAID5 → 実効 6TB
Zone B (2〜4TB):  3台参加 → RAID5 → 実効 4TB
Zone C (4〜6TB):  1台参加 → SINGLE → 実効 2TB (冗長性なし)
```

Zone C は 6TB ディスクしか到達しないため、冗長性を確保できない。

## 検討した選択肢

### 1. 現行方式: SINGLE で追加 + 警告 ✅ 採用

1台しか参加できないゾーンを SINGLE (冗長なし) で作成し、`NoRedundancy` 警告を出す。

- **メリット**: 容量を最大限活用できる。シンプル。
- **デメリット**: 冗長性のない領域にデータが配置される可能性がある。
- **採用理由**: 容量効率を優先。警告により利用者がリスクを認識できる。

### 2. 未使用保留 (Synology SHR 方式)

1台しか参加できないゾーンを作成せず、容量を未使用として保留する。新しいディスクが追加されて冗長化可能になった時点で自動的にゾーンを作成する。

- **メリット**: 冗長性のない領域にデータが置かれることがなく、安全。
- **デメリット**: 最大容量ディスクの末尾が無駄になる。
- **不採用理由**: 容量が無駄になる。利用者がリスクを理解した上で使う方が合理的。

### 3. パーティション分割で RAID 参加

最大容量ディスクの余り部分をパーティションに切り、既存ゾーンの RAID アレイに追加参加させる案。

例: 6TB ディスクの 4-6TB 部分を Zone B に追加参加:

```
Zone B: Disk1[2-4TB], Disk2[2-4TB], Disk3[2-4TB], Disk3[4-6TB]
        → 4メンバー (Disk3 から2パーティション参加)
```

**この方法は容量増加に寄与しない。** 理由:

同一物理ディスクから2パーティションが参加するため、そのディスクが故障すると2メンバー同時喪失する。これに耐えるには RAID レベルを上げる必要がある:

- パーティション追加前: RAID5, N メンバー → 実効 S × (N - 1)
- パーティション追加後: RAID6, N+1 メンバー → 実効 S × (N + 1 - 2) = S × (N - 1)

**パリティ増加分と追加容量が完全に相殺される。** これは一般的に成立する数学的性質であり、同一ディスクからの1パーティション追加は常に効果がない。

注: 複数の異なるディスクからそれぞれ余りパーティションを追加する場合は効果がある可能性があるが、ゾーン方式は既にそのケースを正しく処理している (複数ディスクが同じ容量帯に達すれば自動的にゾーンが構成される)。

### 4. Unraid 方式 (パリティディスク専用)

最大容量ディスクをパリティ計算専用にし、残りのディスクを独立したデータディスクとして運用する方式。

仕組み:
- 最大容量ディスクにデータは置かず、全データディスクの XOR パリティを保存
- 各データディスクは独立したファイルシステム (XFS 等) を持つ
- 任意の1台が故障しても、パリティから復元可能

比較:

| 項目 | puddle (ゾーン方式) | Unraid 方式 |
|------|-------------------|-------------|
| 読み書き性能 | ストライプで高速 | パリティディスクがボトルネック |
| 容量効率 | 突出ディスクの余りが SINGLE | 最大ディスク丸ごとパリティ |
| データ救出 | mdadm + LVM で可能 | 専用ツール必要 |
| 実装複雑度 | mdadm 利用で低い | 自前パリティ管理で高い |
| ディスク独立性 | LVM 結合のため不可 | 各ディスク単体で読める |

不採用理由:
- mdadm を捨てて自前パリティ管理が必要 → 「車輪を再発明しない」の設計思想に反する
- ストライプなしのため性能面で劣る
- 最大容量ディスク丸ごとパリティ専用で、ゾーン方式の部分的 SINGLE より非効率な場合がある
- 大幅なアーキテクチャ変更が必要

### 5. チャンクベース (ZFS / Btrfs 的)

小さなチャンク (例: 1GB) 単位でディスク横断の RAID グループを動的に構成する方式。

- **メリット**: 最も柔軟。容量効率と冗長性を最大化できる。
- **デメリット**: mdadm では実現不可能。ファイルシステムレベルの再実装が必要で、事実上 ZFS/Btrfs の再発明になる。
- **不採用理由**: 実装コストが膨大。既存の成熟したツール (mdadm, LVM) を活用する puddle の設計思想に合わない。

## 結論

ゾーン分割方式には「1台しか参加できないゾーンの冗長性」という本質的な限界がある。これは物理的制約に起因するものであり、mdadm ベースのアーキテクチャでは根本的な解決が困難である。

puddle では以下の方針を採用する:

1. **1台ゾーンは SINGLE で作成し、`NoRedundancy` 警告を出す**
2. 利用者にリスクを明示した上で、容量を最大限活用する
3. 新しいディスク追加で冗長化可能になった場合は、自動的に RAID レベルが昇格する

この方針は容量効率とシンプルさを優先した設計判断である。
